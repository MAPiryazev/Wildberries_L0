# Wildberries_L0
Данный репозиторий хранит домашнее задание уровня L0 техношколы wildberries

# Допилено

**1.**  Теперь большее количество ошибок обработаны с помощью errors.Is(), в main появились блоки switch case которые распознают тип ошибки и сигнализируют о них по разному. Также подключаемые пакеты теперь не падают с log.fatal, а возвращают свои ошибки в main наверх, где уже принимается решение падать или нет. 
**2.** Кэш создается с параметром max int, который управляет размером кеша. Еще кеш после стал не просто мапой, а полноценной структурой LRU кеша - {мапа, мьютекс, двусвязный список}. Если кеш заполнен, и в него кто то добавляет объект, он удалит самый старый и сохранит полученный объект.
**3.** errors.Is(err = context.Cancelled || err = context.DeadlineExceeded) теперь используется 
**4.** Валидация данных теперь реализована в файле validation и в качестве тегов для модели в файле model. Используется **go-playground/validator/v10
**5.** Graceful shutdown реализован со следующим функционалом: Завершается API и прием заказов, завершается consumer кафки, и завершается БД. Сообщения не дошедшие в БД остаются в кафке
**6.** Добавлено несколько unit тестов 
**7.** DLQ реализован как отдельный топик в кафке, если на этапе обработки / получения сервисом заказа возникают ошибки, заказ отправляется в DLQ

# Инструкция по запуску
1. Склонируйте репозиторий себе на компьютер командой git clone https://github.com/MAPiryazev/Wildberries_L0
2. Перейдите в папку репозитория и запустите docker контейнеры командой docker compose up -d
3. Перейдите в папку cmd и запустите сам веб сервис командой go run main.go
После этого шага будет запущена API сервиса, и по адресу localhost:8081 в браузере отобразится страница для получения данных заказаа по его uid (изначально в БД будет только 1 заказ с uid b563feb7b2b84b6test)
4. Далее чтобы запустить продьюсера сообщений который будет пересылать заказы в kafka нужно:
    4.1. по пути Wildberries_L0/internal/kafka/producer/test_jsons запустить генератор фейковых заказов командой go run generator.go и подождать какое-то время, можно прервать его самостоятельно т.к. он генерит 3 млн заказов
    4.2. запустить producer.go в папке producer (после того как уже запущен сам сервис)

# Кэш
Кэш в данном проекте представляет собой мапу (в соответствии с ТЗ), которая заполняется в 3х случаях:
1) При запуске сервиса кэш получает определенное количество последних созданных заказов
2) При добавлении нового заказа в БД
3) При запросе пользователем 

С учетом того что БД не индексирована, использование кэша на несколько порядков увеличивает скорость выдачи данных о заказе пользователю. Ниже представлен логарифмический график для сравнения подрядков скорости выдачи заказа пользователю с использования кэша и без него. Во время тестирования в БД было окооло 3-х млн заказов.

# БД
В коде для работы с подключениями к БД используются connection pools.
Модель данных - снежинка
+ Меньше дублирование данных
+ Легче поддерживать изменения таблиц, т.к. они не сильно связаны

- В запросах много join-ов, это так же увеличивает нагрузку на БД
